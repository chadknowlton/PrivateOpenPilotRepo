3. Cross Site Request Forgery  [State: Not Applicable]  [Priority: High] 

Category:Elevation Of Privilege
Description:Cross-site request forgery (CSRF or XSRF) is a type of attack in which an attacker forces a user's browser to make a forged request to a vulnerable site by exploiting an existing trust relationship between the browser and the vulnerable web site. In a simple scenario, a user is logged in to web site A using a cookie as a credential. The other browses to web site B. Web site B returns a page with a hidden form that posts to web site A. Since the browser will carry the user's cookie to web site A, web site B now can take any action on web site A, for example, adding an admin to an account. The attack can be used to exploit any requests that the browser automatically authenticates, e.g. by session cookie, integrated authentication, IP whitelisting. The attack can be carried out in many ways such as by luring the victim to a site under control of the attacker, getting the user to click a link in a phishing email, or hacking a reputable web site that the victim will visit. The issue can only be resolved on the server side by requiring that all authenticated state-changing requests include an additional piece of secret payload (canary or CSRF token) which is known only to the legitimate web site and the browser and which is protected in transit through SSL/TLS. See the Forgery Protection property on the flow stencil for a list of mitigations.
Justification:Does not make sense in this context; The connection between the Comma device and the Openpilot logging service is offline and not through a browser

4. Elevation by Changing the Execution Flow in 1.0 Openpilot Logging Service  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:An attacker may pass data into 1.0 Openpilot Logging Service in order to change the flow of program execution within 1.0 Openpilot Logging Service to the attacker's choosing.
Justification:The Panda safety module monitors information passed back from the device to Openpilot to prevent output that isn&#39;t properly formatted. The only information Panda is supposed to hand back to Openpilot is logging information regarding the vehicle state, so injected code would be caught and rejected by the safety module

5. Potential Data Repudiation by 1.0 Openpilot Logging Service  [State: Not Applicable]  [Priority: High] 

Category:Repudiation
Description:1.0 Openpilot Logging Service claims that it did not receive data from a source outside the trust boundary. Consider using logging or auditing to record the source, time, and summary of the received data.
Justification:Due to the real-time aspect of the comma device dataflow, occasionally dropped data is to be expected

6. 1.0 Openpilot Logging Service May be Subject to Elevation of Privilege Using Remote Code Execution  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:Comma Device may be able to remotely execute code for 1.0 Openpilot Logging Service.
Justification:Data sent from the comma device to the openpilog logging service contains log data and not ececutable code. If executable code were to be passed, The Panda Safety package would detect and prevent the execution of the code.

7. Potential Process Crash or Stop for 1.0 Openpilot Logging Service  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:1.0 Openpilot Logging Service crashes, halts, stops or runs slowly; in all cases violating an availability metric.
Justification:Openpilot has an overhead manager process which handles the starting and stopping of all its subprocesses, and processes which crash or stop during runtime will be restarted

8. Data Flow Comma Device Data Is Potentially Interrupted  [State: Not Applicable]  [Priority: High] 

Category:Denial Of Service
Description:An external agent interrupts data flowing across a trust boundary in either direction.
Justification:The connection between external Comma device components such as the sensors and the Openpilot software is encapsulated within the device itself, so physically interrupting the connection between the two would require opening up the hardware

9. Data Flow Sniffing  [State: Mitigation Implemented]  [Priority: High] 

Category:Information Disclosure
Description:Data flowing across Comma Device Data may be sniffed by an attacker. Depending on what type of data an attacker can read, it may be used to attack other parts of the system or simply be a disclosure of information leading to compliance violations. Consider encrypting the data flow.
Justification:This event is highly unlikely and requires physical compromise of OpenPilot. This transfer of data is local and is not externally facing for any sort of external entity to compromise.

10. Potential Lack of Input Validation for 1.0 Openpilot Logging Service  [State: Mitigation Implemented]  [Priority: High] 

Category:Tampering
Description:Data flowing across Comma Device Data may be tampered with by an attacker. This may lead to a denial of service attack against 1.0 Openpilot Logging Service or an elevation of privilege attack against 1.0 Openpilot Logging Service or an information disclosure by 1.0 Openpilot Logging Service. Failure to verify that input is as expected is a root cause of a very large number of exploitable issues. Consider all paths and the way they handle data. Verify that all input is verified for correctness using an approved list input validation approach.
Justification:The Comma device sanitizes the data it outputs, mitigating the risk of input arriving not as expected
